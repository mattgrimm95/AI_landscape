# /mvp-build

You are about to build a complete MVP in one sustained pass. This is a deliberately-chosen pattern (the user override of expert consensus, recorded in `skills_plan.md`). Your job is to make the one-shot as safe as possible by following the guardrails below.

## Workflow

### Step 1 — Read and verify the spec

1. Read the spec. Default path: `./mvp-spec.md`. If a different path was passed as argument, use it.
2. If the file doesn't exist: tell the user to run `/brainstorm` first, then stop.
3. **Verify all twelve decisions are answered.** If any section is empty, says "TBD," or is obviously a placeholder, stop and ask the user to complete it via `/brainstorm`. Do not improvise around an ambiguous spec — that's the failure mode this skill exists to prevent.

### Step 2 — Build the walking skeleton AS ITS OWN COMMIT

Before anything else, commit a minimal end-to-end slice. This is the revertibility safety net.

The skeleton must include:
- **One source-of-truth file** with exactly ONE sample row matching the schema from spec decision #2.
- **One CLI verb** named `stats` that reads the file and prints a count (e.g. `1 row`).
- **One API endpoint** (only if API is in spec decision #4) returning the count as JSON.
- **One approval test** that pins the `stats` output.
- **One CI workflow** (`.github/workflows/tests.yml`) that runs the test on push.
- **One README** with the Mermaid diagram and one-line install + `stats` invocation.

After writing, **run the test once** to prove it passes. If it fails, fix it before continuing — don't add more surface on top of a broken foundation.

Commit message:
```
walking skeleton: <project> end-to-end with one row

CLI: stats
API: GET /api/count (if applicable)
Test: tests/test_stats.py (approval)
CI: .github/workflows/tests.yml
```

### Step 3 — Build the full MVP in the same session

Each subsystem gets its own commit so the history reads as a build log. Order:

1. **Full source-of-truth schema** + canonical example fixtures (~3-5 rows covering common cases).
2. **Full CLI surface** — all verbs from spec decision #4. Each verb gets a docstring and `--help` text.
3. **Full API surface** (if applicable) — all endpoints from spec decision #4. Use FastAPI's `TestClient` for tests so the test suite bypasses uvicorn (and the bind-address class of bugs).
4. **Web UI** (if applicable) — vanilla JS + CDN libraries (Cytoscape / D3 / etc.), **no build step**. One HTML file + one CSS file + one JS file is the default shape.
5. **Test suite reaches ~30 tests** covering:
   - **approval tests** (Llewellyn Falco, https://approvaltests.com/) on representative outputs — pin actual emitted bytes / structures.
   - **property-based tests** (Hypothesis if Python) for any subsystem with invariants over a large input space — parsing, dedup, reconciliation, idempotence checks.
   - **adversarial battery** for the noisiest subsystem per spec decision #10 — code blocks, math notation, malformed feeds, missing fields, whatever applies.
   - **no-secrets guard test** — scans the repo for AWS / Anthropic / OpenAI / GitHub / Google / Slack / private-key patterns. ALWAYS included regardless of whether the spec mentions secrets.
   - Comment each test with which strategy it represents (e.g. `# approval`, `# property`, `# adversarial`, `# no-secrets`).
6. **`DECISIONS_LOG.md`** with a MADR-format seed entry (Context / Decision / Consequences) recording the build session. Reference the brainstorm spec.
7. **`llms.txt` + `llms-full.txt`** — autogenerated from public function signatures. For Python: ast-walk modules, emit one line per public function with signature + first docstring line. `llms.txt` is the index; `llms-full.txt` is the full content. **Do NOT create `LLM_INDEX.md`** (legacy).
8. **`docs/getting-started.md`** (Diátaxis tutorial) + `docs/recipes/` (how-to guides skeleton) + README (reference + explanation). README references the recipes.
9. **`.github/workflows/tests.yml`** — matrix Python 3.9 / 3.11 / 3.12 (or equivalent for the project's language), **no-secrets guard job runs first**.
10. **`.github/workflows/docker.yml`** (if container is in spec) — build verify on every PR + push, build-and-push to GHCR on main, **smoke-test-published-image job** that pulls the just-published image by SHA tag, runs it, polls a real endpoint for HTTP 200 within 60s. The smoke-test step's existence catches the class of bugs (bind address, missing files, broken startup) that unit tests can't see.
11. **`Dockerfile`** (multi-stage, non-root user, healthcheck, derived DBs pre-built so the image ships ready-to-serve) + **`docker-compose.yml`** (bind-mounts the derived-data dir for persistence) if container is in spec.

### Step 4 — Final message to the user

```

- Walking-skeleton checkpoint at commit {sha-of-skeleton}
- Full build at HEAD

This is a STARTING POINT, not a final implementation. The expert-consensus
posture (Willison, Beck, et al.) is that LLM-generated code is "the first
draft." Iterate now:

  1. Run the app: {invocation from README}
  2. Jot numbered notes on what's wrong, missing, or surprising
  3. Run `/feedback-triage` to convert them into a fix list

The first thing you'll catch is the brainstorm decision you wish you'd
answered differently. That's expected. Loop back to /brainstorm if a
decision needs rethinking; otherwise /feedback-triage is the right next
move.
```

## Rules (read these BEFORE building)

- **Modular monolith.** One package, separately-testable modules (e.g. `<package>/{scraper,storage,ner,api,cli}`). NEVER microservices at MVP.
- **CLI-first.** Every capability is a CLI verb. The web app calls the same modules. Tests bypass the web layer via `TestClient`.
- **Minimize deps; justify each one.** Stdlib when the abstraction is genuinely simple. Reject deps that trade simplicity for convenience. The spec's deps-allowance list (decision #5) is the upper bound.
- **Non-destructive defaults.** Append-only writes. Archive-don't-delete sidecar for any pruning operation. The one destructive verb requires `--confirm`.
- **Bind addresses in containers.** If the spec has a web server AND a container, the Dockerfile CMD must pass `--host 0.0.0.0`. The CLI default stays `127.0.0.1`. (This catches a known LLM failure mode where the server binds loopback and Docker port-forwarding never reaches it.)
- **Cache-first for LLM features.** If spec decision #7 says "yes, API keys involved," implement the sidecar snapshot pattern: `snapshots/<feature>/YYYY-MM-DD.json`, `/api/<feature>` reads cache first, `?refresh=1` regenerates when a key is present.
- **Tests by strategy, not by count.** The ~30 figure is rough — what matters is that approval + property + adversarial + no-secrets are each represented.
- **`DECISIONS_LOG.md` uses MADR.** Context / Decision / Consequences sections. NOT "Why / Change / Verification."
- **`CLAUDE.md` stays short** (load-once-per-session). The function index belongs in `llms-full.txt`.
- **DO NOT push to a remote.** Commit locally; let the user push when they've reviewed.
- **DO NOT execute any destructive action.** No `git push --force`, no `rm -rf`, no DB drops. Even if it would speed things up.
- **STOP and ask** if you hit any ambiguity in the spec or a blocker you can't resolve (dep install fails, an external service is unreachable, etc.). Do NOT improvise.
